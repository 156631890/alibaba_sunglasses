# Gemini Listing Generation Design

Date: 2026-05-19

## Goal

Add optional Gemini-powered English listing generation to the sunglasses listing MVP. The user should keep working in the same Chinese workbench, but can click one extra button to replace the deterministic template output with richer English listing content generated by Gemini.

The feature must be a controlled enhancement, not a dependency. If Gemini is not configured, times out, returns invalid JSON, or refuses a prompt, the app must keep the current template-generated listing usable.

## Product Behavior

The app keeps the existing workflow:

1. User fills or fetches product information.
2. User clicks `分析产品`.
3. User clicks `生成上架内容` to create the deterministic fallback listing.
4. User can click `Gemini 生成英文内容` to request AI-enhanced English copy.
5. On success, the right output panel is replaced with the Gemini listing package.
6. On failure, the current listing stays unchanged and the UI shows a Chinese error message.

All UI labels, status messages, and errors remain Chinese. All generated listing content remains English.

## Recommended Approach

Use a backend API proxy:

- Frontend calls `POST /api/generate-listing`.
- Vercel serverless function reads `GEMINI_API_KEY`.
- Server calls Gemini REST `generateContent`.
- Server asks for JSON output with the same `ListingPackage` shape used by the existing app.
- Server validates and normalizes the Gemini response before sending it to the browser.

This avoids exposing the Gemini API key in frontend JavaScript and keeps the browser code provider-agnostic.

## Official API Assumptions

The implementation should use the Gemini REST API documented by Google:

- Text generation: `generateContent`
- API key header: `x-goog-api-key`
- JSON generation support through `responseMimeType: "application/json"`
- Optional model environment override through `GEMINI_MODEL`

Docs checked:

- `https://ai.google.dev/api`
- `https://ai.google.dev/gemini-api/docs/text-generation`
- `https://ai.google.dev/gemini-api/docs/structured-output`

## Environment Variables

Required for AI generation:

```text
GEMINI_API_KEY=...
```

Optional:

```text
GEMINI_MODEL=gemini-2.5-flash
```

If `GEMINI_MODEL` is missing, the server uses `gemini-2.5-flash`.

No Gemini key should ever be stored in source code, committed files, generated CSV, browser local storage, or frontend environment variables.

Endpoint:

```text
POST /api/generate-listing
```

Request:


## Response Shape

The generated result must match the existing `ListingPackage` type exactly:

- `productTitle`
- `b2bTitle`
- `slug`
- `metaTitle`
- `metaDescription`
- `htmlDescription`
- `faq`
- `imageAltText`
- `seoKeywords`
- `geoSummary`
- `schemaJson`
- `imagePrompts.whiteBackground`
- `imagePrompts.lifestyle`
- `imagePrompts.detail`
- `imagePrompts.customization`
- `imagePrompts.shortVideo`

This lets copy buttons and CSV export keep working without a separate AI output model.

## Prompt Requirements

The server prompt must instruct Gemini to:

- Generate English ecommerce and B2B listing content only.
- Keep UI-independent output; do not include Chinese in listing fields.
- Avoid third-party brand names, replica language, lookalike wording, celebrity references, platform names, watermarks, and claims not supported by the input.
- Preserve safe product facts from `input`, `analysis`, and `fallbackListing`.
- Prefer generic sunglasses buyer language for international wholesale, OEM, ODM, and retail contexts.
- Return valid JSON only, with no Markdown fences or explanatory text.

The prompt should include the risk matches and replacements from `analysis.risk.matches`. If risk level is `High`, Gemini should still generate safe generic content, but the frontend should keep showing the high-risk analysis.

## Server Validation

The server should validate Gemini output before returning it:

- Parse JSON safely.
- Require all top-level `ListingPackage` fields.
- Require all five `imagePrompts` fields.
- Ensure array fields are arrays of strings.
- Ensure `schemaJson` parses as JSON; if it does not, replace it with the deterministic fallback schema.
- Regenerate `slug` from `productTitle` if the model returns an empty or unsafe slug.
- Clamp `metaTitle` to 60 characters and `metaDescription` to 156 characters.
- Reject output containing risky terms already covered by the local risk rules.

If validation fails, return a structured error and do not replace the current listing in the UI.

## Error Handling

Structured error codes:

- `GEMINI_NOT_CONFIGURED`: `GEMINI_API_KEY` is missing.
- `GEMINI_REQUEST_FAILED`: Gemini API returned non-2xx or the network failed.
- `GEMINI_TIMEOUT`: request exceeded the server timeout.
- `GEMINI_INVALID_RESPONSE`: response was not valid JSON or failed shape validation.
- `METHOD_NOT_ALLOWED`: request was not POST.

The UI should show these messages in Chinese and keep the current listing unchanged.

## Local Development

The local Vite dev server should support the same route pattern as the product fetch API:

- `vite.config.ts` should mount `/api/generate-listing` in local dev.
- The local route should call the same server helper used by the Vercel endpoint.
- If `GEMINI_API_KEY` is not available locally, the route should return `GEMINI_NOT_CONFIGURED` instead of failing with a generic exception.

This makes local testing match Vercel behavior.

## UI Changes

Add a second generation button near the existing listing action:

- Existing `生成上架内容`: deterministic fallback.
- New `Gemini 生成英文内容`: AI-enhanced generation.

The new button should be disabled while a Gemini request is running. The output panel should show a concise Chinese status:

- Loading: `Gemini 正在生成英文上架内容...`
- Success: `Gemini 已生成英文内容，请检查后再发布。`
- Not configured: `Gemini API Key 未配置，请先添加 GEMINI_API_KEY。`
- Failure: use the API error message.

The generated English content should appear in the same output fields as the deterministic listing. CSV export should export whichever listing is currently displayed.

## Testing

Automated tests:

- Server helper returns `GEMINI_NOT_CONFIGURED` when no key is present.
- Server helper sends a valid Gemini request body when a key exists.
- Server helper parses valid Gemini JSON into a `ListingPackage`.
- Server helper rejects invalid JSON.
- Server helper rejects missing required fields.
- Server helper rejects known risky terms in model output.
- Frontend keeps the existing listing unchanged when Gemini fails.

Manual verification:

- Run without `GEMINI_API_KEY` and confirm the Chinese not-configured error.
- Run with a test key and confirm `Gemini 生成英文内容` replaces output with English listing content.
- Confirm deterministic `生成上架内容` still works without Gemini.
- Confirm CSV export uses the currently displayed listing.
- Run `npm test -- --run`.
- Run `npm run build`.
- Run standalone type checks for Vercel API files if Vite build does not cover them.

## Deployment Notes

Vercel environment variables must include:

```text
GEMINI_API_KEY
```

Optional:

```text
GEMINI_MODEL
```

`vercel.json` must continue excluding `/api/*` from SPA rewrites so both `/api/fetch-product` and `/api/generate-listing` remain reachable.

## Acceptance Criteria

- The app can still generate deterministic English listing content without Gemini.
- When Gemini is configured, the user can generate richer English listing content from the same form.
- The Gemini API key is never exposed to frontend code.
- All UI labels and errors remain Chinese.
- All generated listing fields remain English.
- Risky brand, replica, lookalike, celebrity, watermark, and platform-source terms are not returned in generated listing copy.
- The output keeps the existing `ListingPackage` shape, so copy buttons and CSV export continue working.
- Tests and production build pass.
